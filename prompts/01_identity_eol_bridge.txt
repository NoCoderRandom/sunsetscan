SESSION 01: Bridge Device Identification into EOL Checking
============================================================

READ THE SESSION LOG FIRST: SESSION_LOG.md

CONTEXT:
SunsetScan has two pipelines that currently don't talk to each other:

  Pipeline A — EOL checking (check_eol_status in sunsetscan.py)
    Runs BEFORE device identification. Reads port_info.service and
    port_info.version from nmap. Maps via eol/product_map.py PRODUCT_MAP
    to an endoflife.date slug. Only works when nmap reports a clean
    service+version like "nginx/1.18.0" or "OpenSSH_8.2".

  Pipeline B — Device identification (run after security checks)
    Fuses 14 extractors to determine vendor="Synology", model="DS920+",
    version="7.2", device_type="NAS". This rich result is NEVER fed to
    the EOL checker.

  What gets missed:
    - Synology NAS detected via HTTP fingerprint as DSM 7.2 — EOL checker
      never checks synology-dsm 7.2 because nmap just sees "http"
    - MikroTik SSH banner says RouterOS 6.48 — EOL checker sees "ssh"
    - QNAP QTS, pfSense, Proxmox VE, VMware ESXi — same problem
    - Any firmware version from HttpFingerprint.firmware_version

  Two existing bridge pipelines already exist as examples:
    - _run_ja3s_eol_pipeline() chains JA3S matches -> EOL
    - _run_snmp_eol_pipeline() chains SNMP sysDescr -> EOL
  We need a third bridge for device identity -> EOL.

TASK:

1. CREATE _run_identity_eol_pipeline() IN sunsetscan.py
   After the device identification loop completes in run_security_checks(),
   iterate self.last_device_identities and for each identity that has a
   vendor+version, attempt an EOL check:

   a) Build a mapping from DeviceIdentity fields to PRODUCT_MAP slugs:
      - Use vendor.lower() as a key into PRODUCT_MAP (e.g. "synology" -> "synology-dsm")
      - Also try "{vendor} {device_type}".lower() for more specific matches
      - Also try model name as a key (e.g. "routeros" -> "mikrotik")
      - Import get_product_slug() from eol.product_map for the lookup

   b) If a slug is found AND the identity has a version string:
      - Call self.eol_checker.check_version(slug, version)
      - Store result in self.last_eol_data[ip][0] (port 0 = firmware-level)
        Use port 0 as a convention for "device firmware" not tied to a port.

   c) Generate findings for firmware EOL:
      - EOL firmware: Severity.HIGH — "Firmware end-of-life: {vendor} {version}"
      - Approaching EOL (< 6 months): Severity.MEDIUM
      - Supported: no finding (or INFO if you want)
      - Add to self.finding_registry

   d) Log a debug line for each identity->EOL mapping attempted

   IMPORTANT: This runs AFTER device identification, so identity data is
   already in self.last_device_identities. Do NOT re-run identification.

2. ALSO CHECK HttpFingerprint.firmware_version DURING check_eol_status()
   In the existing check_eol_status() method (around line 693), after the
   banner/service check, add a fallback:

   - If no EOL result was found via banner/service, check if the port has
     an http_fingerprint with a firmware_version
   - If it does, try to map the fingerprint's device_type to a product slug
     and check the firmware_version against EOL
   - Example: HttpFingerprint(device_type="TP-Link Router", firmware_version="3.0.0.4")
     -> try get_product_slug("tp-link") -> check version "3.0.0.4"

3. UPDATE DEVICE ID TABLE TO SHOW EOL STATUS
   In _print_device_id_table() (sunsetscan.py), add an "EOL" column:

   - After the identification table is built, check if each IP has a
     port-0 (firmware) EOL entry in self.last_eol_data
   - If EOL: show "[red]EOL[/red]" with the date
   - If approaching: show "[yellow]Soon[/yellow]"
   - If supported: show "[green]OK[/green]"
   - If no data: show "—"

4. UPDATE HTML REPORT DEVICE INVENTORY TABLE
   In ui/templates/report.html.j2, the device inventory table should show
   an EOL column too. The data needs to flow through:

   a) In sunsetscan.py, when calling export_html(), the last_eol_data dict
      already gets passed. The template just needs to cross-reference
      device_identities[ip] with eol_data[ip][0] (firmware EOL).

   b) In the Jinja template's device inventory section, add a column:
      - "Firmware EOL" with colored badge (red/yellow/green)
      - Show the EOL date and days remaining if available

5. ADD IDENTITY->EOL MAPPING TABLE
   Create a small mapping dict in sunsetscan.py (or a helper module) that
   maps common device identity vendor names to their PRODUCT_MAP slugs:

   _IDENTITY_EOL_MAP = {
       "synology": "synology-dsm",
       "qnap": "qnap-qts",
       "mikrotik": "mikrotik",         # NOT_TRACKED but map anyway
       "netgate": "pfsense",
       "proxmox": "proxmox-ve",
       "vmware": "vmware-esxi",
       "cisco": "cisco-ios-xe",
       "juniper": "junos",
       "canonical": "ubuntu",           # Ubuntu servers
       "red hat": "rhel",
       "microsoft": "windows-server",
   }

   This supplements get_product_slug() for cases where the vendor name
   alone is enough to identify the product (e.g. Synology = DSM, always).
