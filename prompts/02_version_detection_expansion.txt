SESSION 02: Version Detection Expansion + SNMP + Scan Profiles
================================================================

READ THE SESSION LOG FIRST: SESSION_LOG.md

CONTEXT:
After prompt 01 bridges device identification into EOL checking, the next
bottleneck is version detection itself. Three problems remain:

  A) QUICK scan profile uses "-T4 -F" — no -sV flag, so nmap returns empty
     service/version fields for most ports. Banner grabber partially
     compensates but the EOL checker gets very little to work with. This is
     the DEFAULT profile, so most users hit this.

  B) SNMP sysDescr patterns in core/snmp_checker.py only recognize 15
     firmware/OS signatures. Missing: Fortinet FortiOS, WatchGuard Fireware,
     SonicWall SonicOS, Aruba AOS, Huawei VRP, HP iLO, Dell iDRAC, TrueNAS,
     OPNsense, and others that commonly expose SNMP.

  C) SSH banners like "SSH-2.0-OpenSSH_8.2p1 Ubuntu-4ubuntu0.5" contain
     both the OpenSSH version AND the embedded OS distribution+version.
     Currently the device identifier uses this for type classification but
     the version info is not piped to EOL checking. Neither the EOL checker
     nor the JA3S/SNMP pipelines extract the embedded OS version.

TASK:

1. FIX QUICK SCAN PROFILE
   In config/settings.py, change the QUICK profile to include version
   detection:

   Current:  "QUICK": "-T4 -F"
   Change:   "QUICK": "-T4 -F -sV --version-intensity 2"

   --version-intensity 2 (instead of default 7) keeps it fast while still
   grabbing the most common service banners. This is a minimal change with
   large impact — every default scan will now get version data.

   Also update the SCAN_DESCRIPTIONS dict to note that QUICK now includes
   light version detection.

   Also update the README.md scan profiles table (search for "QUICK") to
   reflect the new flags.

2. EXPAND SNMP sysDescr PATTERNS
   In core/snmp_checker.py, add patterns to _SYSDESCR_PATTERNS for:

   - FortiGate / FortiOS:
     Pattern: r'FortiGate.*?v(\d+\.\d+\.\d+)' or r'FortiOS[- ]v?(\d+\.\d+(?:\.\d+)?)'
     Slug: 'fortios'

   - WatchGuard Fireware:
     Pattern: r'WatchGuard.*?Fireware[/ ]v?(\d+\.\d+(?:\.\d+)?)'
     Slug: 'watchguard'  (note: NOT_TRACKED, but map for device ID)

   - SonicWall SonicOS:
     Pattern: r'SonicOS[- ](\d+\.\d+(?:\.\d+)?)'
     Slug: 'sonicwall-sma'

   - Aruba AOS / ArubaOS:
     Pattern: r'ArubaOS[- ](\d+\.\d+(?:\.\d+)?)'
     Slug: 'arubaos'  (add to PRODUCT_MAP too)

   - Huawei VRP:
     Pattern: r'Huawei.*?VRP.*?V(\d+R\d+(?:C\d+)?)'  (Huawei uses V200R003C00 format)
     Slug: 'huawei-vrp'  (add to NOT_TRACKED)

   - HP iLO:
     Pattern: r'iLO\s*(\d+)\s+v?(\d+\.\d+)' -> slug 'hp-ilo' (use group 2 for version)
     (add to NOT_TRACKED)

   - Dell iDRAC:
     Pattern: r'iDRAC[- ]?(\d+)' -> version from group 1
     Slug: 'idrac'  (add to NOT_TRACKED)

   - TrueNAS / FreeNAS:
     Pattern: r'(?:TrueNAS|FreeNAS)[- ](\d+\.\d+(?:\.\d+)?)'
     Slug: 'freenas'

   - OPNsense:
     Pattern: r'OPNsense[- ](\d+\.\d+(?:\.\d+)?)'
     Slug: 'opnsense'

   - Windows Server (from SNMP):
     Pattern: r'Windows Server (\d{4}(?:\s+R2)?)'
     Slug: 'windows-server'

   Also add the new slugs to eol/product_map.py:
   - Add to PRODUCT_MAP: "arubaos" -> "arubaos", "huawei vrp" -> "huawei-vrp",
     "hp ilo" -> "hp-ilo", "idrac" -> "idrac"
   - Add to NOT_TRACKED_PRODUCTS: "huawei-vrp", "hp-ilo", "idrac", "watchguard",
     "arubaos" (if not on endoflife.date — verify by checking the URL format
     https://endoflife.date/api/{slug}.json in a comment)

3. CREATE SSH BANNER -> EOL PIPELINE
   In sunsetscan.py, create _run_ssh_eol_pipeline():

   After per-host security checks complete (but within the same host processing),
   iterate host_info.ports for SSH services and extract embedded OS info:

   a) Parse SSH banners for OS distribution:
      Patterns to extract:
      - r'OpenSSH.*Ubuntu[- ](\d+)(?:ubuntu[\d.]+)' -> slug "ubuntu", version from group 1
        (e.g. "OpenSSH_8.2p1 Ubuntu-4ubuntu0.5" -> ubuntu 8? No!)
        Actually: the Ubuntu version is NOT in the SSH banner directly.
        The "4ubuntu0.5" is the package revision, not the OS version.
        HOWEVER: the Debian package version does hint at the OS release:
        - "Ubuntu-4ubuntu0" = Ubuntu 20.04
        - "Ubuntu-7ubuntu0" = Ubuntu 22.04
        This mapping is fragile. Better approach: just use the OpenSSH
        version itself for EOL checking (OpenSSH is on endoflife.date? No,
        it's in NOT_TRACKED). So skip this — OpenSSH is not tracked.

      Better approach — extract the OS name for the device identifier only,
      and the service version for CVE checking:
      - r'OpenSSH_([\d.p]+)' -> product "openssh", version for CVE lookup
      - r'dropbear_([\d.]+)' -> product "dropbear", version for CVE lookup

   b) In _check_single_host(), after the SSH checker completes, extract
      the SSH service version and feed it to the CVE checker:
      - Check if port_info.banner contains "OpenSSH_X.Y" or "dropbear_X.Y"
      - If so, call self.cve_checker.check() with product="openssh" version="X.Y"
      - This closes the gap where SSH versions are visible but not CVE-checked

   c) For OS distribution detection from SSH, instead add patterns to the
      device_identifier's _extract_from_ssh_banner:
      - r'OpenSSH.*Ubuntu' already exists (maps to Linux Server 0.25)
      - ALSO extract the package version suffix and map to Ubuntu release:
        "4ubuntu0" -> version hint "20.04", "7ubuntu0" -> "22.04"
        Add this as a version field in the evidence (low confidence 0.15)

   NOTE: Be careful here. The SSH banner OS extraction is inherently
   unreliable. The primary goal is CVE checking for the SSH daemon itself,
   not OS version detection. Keep OS hints at very low confidence.

4. EXTRACT HTTP FINGERPRINT VERSION FOR CVE CHECKING
   In _check_single_host() or _run_cve_checks(), add a step:

   - After regular CVE checks on port_info.service + port_info.version,

   - This catches cases like "Apache/2.4.29" in raw_headers that nmap
